commy plugin: release 0.20.0

Minor release of the commy plugin, **0.19.1 → 0.20.0**.

This release changes the behaviour of `post` against a resolved thread. It is a
bug fix, but it is a bug fix you must sequence — read **Upgrading** before you
take it.

## Highlights

**`post` no longer forks a resolved thread into a sibling topic.** Posting to a
thread by name after that thread had been resolved silently created a *new*
topic under the bare name instead of appending to the original. The conversation
split at the resolve, and the caller got back a well-formed `MessageRef`
pointing at the wrong topic — so nothing looked wrong at the call site. Zulip has
no resolved flag; resolution *renames* the topic to `✔ <name>`, and Zulip creates
topics implicitly on send. Two of commy's three topic-addressing paths already
knew that (`read_thread`, `resolve_thread`); the write path did not, so it
addressed a name that no longer existed and Zulip helpfully created it.

`post` now probes for the thread's current substrate form — the same probe
`resolve_thread` already used — and addresses that. The probe tries the plain
name first and the `✔` form second, matching `read_thread`, so the two verbs can
never disagree about which conversation a name refers to (in a realm this bug has
already forked, both forms exist, and a different precedence would have `post`
writing to one and `read_thread` reading the other).

**The behaviour change: a post into a resolved thread appends to it, and the
thread stays resolved.** `MessageRef.resolved` now reports what the probe found
rather than a hardcoded `false`. **`post` never mutates thread state** — if you
want a reply to re-open a thread, call `unresolve_thread` yourself. Auto-unresolve
on post was considered and rejected: it would make `post` a two-request mutating
verb on the substrate's hottest write path, one that can partially fail (the
message lands, the unresolve doesn't).

**Seats no longer hang on their first `post`/`react` after an unlucky resume.**
A returning ephemeral seat reports its resume verdict — "did my event queue
survive, or do I need history catch-up?" — off its first `/events` poll. A poll
that reached neither a success nor a dead-queue answer (a response-decode error,
a non-`BAD_EVENT_QUEUE_ID` API error, a wedged `429` loop) retried forever under
the never-give-up reconnect schedule, so the verdict was never reported. The
seat's first attribution-producing call awaits that verdict inline, so it hung
indefinitely. A bounded 60s fallback now guarantees the latch always completes,
falling back to the duplicate-tolerant history catch-up. A healthy poll returns
in well under a second and always wins the race; the retry loop underneath is
untouched.

## Upgrading

**If any of your flows treat "unresolved" as "needs attention", make the
caller-side change *before* you take this release, not after.**

The old fork was **loud**: you got two topics and a visibly wrong conversation.
The new append is **quiet**: the item simply stops asking. A flow that relied on
"a reply re-opens the thread" — where the fork happened to leave an unresolved
bare-name topic that *looked* like an open item — will, on this build, post into
the resolved thread and leave it resolved. You trade a visible bug for an
invisible one. The fix is to call `unresolve_thread` explicitly wherever a reply
is meant to re-open the item; do that first, then upgrade.

**Note that the plugin's launcher is not version-pinned.** `.mcp.json` launches
`npx -y @codeforbreakfast/commy-mcp`, which resolves npm's `latest` at every MCP
start — so seats pick this release up on their next restart without an explicit
opt-in. If you need to gate the upgrade behind your own caller-side change, do it
with npm's release-age soak (`COMMY_NPM_MIN_RELEASE_AGE`, which this plugin
threads into its own `npx` launch as `npm_config_min_release_age`) rather than
assuming a restart won't take it.
