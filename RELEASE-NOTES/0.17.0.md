Minor release of the commy plugin, **0.16.0 → 0.17.0**.

## Highlights

**A resumed session restores its full subscription set with zero agent action.**
When a seat's process restarts on the same session — a crash, a respawn, a
context reset — the agent used to come back deaf: its subscriptions lived only in
the memory of the process that died. This release makes them durable. Each
subscribe/unsubscribe is now persisted to an on-disk store keyed by session, and
on resume the agent reconstructs its `SessionId` from the boot-injected session
id and reloads its entire persisted narrow set — channels, threads, `mentions` —
before it does anything else. A resumed worker is reachable again immediately,
with no need to re-run `subscribe`. This is what 0.16.0's resume note reached for
but did not actually persist; 0.17.0 is the working spine.

**Precise replay: messages that arrived while a seat was down are delivered on
resume, exactly once.** Restoring the subscription set is only half the story —
the other half is not losing (or duplicating) what was posted during the gap. A
per-`(session, target)` delivery cursor now records how far each subscription has
been delivered, so on resume the agent replays exactly the messages it missed for
its restored narrow set and no more. A seat that was down for a reaction or a
thread reply comes back and receives it, rather than silently skipping it or
re-reading traffic it already saw.

## Maintenance

- The `DeliveryTarget` type is now derived from its Schema, keeping the runtime
  contract and the static type in lockstep (internal; no behaviour change).

**Upgrading:** no action required. Subscription persistence and resume restore
happen automatically; a resumed agent rebuilds its narrow set on its own.
