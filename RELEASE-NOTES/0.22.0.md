commy plugin: release 0.22.0

Minor release of the commy plugin, **0.21.0 → 0.22.0**.

One fix, with a behaviour change behind it: the plugin stops overwriting
`COMMY_*` values you set by other means. If you have ever set `COMMY_SUBSCRIBE`
in a systemd unit, a pane env, or a nix module and found your seat behaving as
though it had no subscriptions, that was this — and taking this release is the
first of two steps to end it. Read **Upgrading** before you assume the fix is
live on your seats; it needs the plugin pin moved as well as the server
published.

## Highlights

**The plugin no longer clobbers `COMMY_SUBSCRIBE` and
`COMMY_CATCHUP_WINDOW_SECONDS` that you set elsewhere.** `.mcp.json` is static
JSON with no way to omit a key conditionally, so every `${user_config.KEY}` it
declares gets written into the MCP child's environment whether or not you
supplied that key — and an unsupplied optional field lands as an empty string.
Written to the bare name, that empty string *overrode* whatever your unit file,
pane env, or nix module had set. The plugin then read "no subscriptions" and
booted a seat that was silently deaf to every channel it was meant to watch. No
error, no warning, nothing in a log to find: an operator who set the variable
correctly and an operator who never set it produced identical output.

The manifest now writes its substitutions to its own key space —
`COMMY_SUBSCRIBE_USER_CONFIG`, `COMMY_CATCHUP_WINDOW_SECONDS_USER_CONFIG` — and
the server reads suffixed-first, falling back to the bare inherited name when
the suffixed one is empty. The two supply paths compose instead of colliding,
and the plugin can only ever overwrite a name it alone owns. Precedence runs
that direction deliberately, because the server floats to npm latest on every
seat start while the manifest moves only when someone bumps the pin: new server
with old manifest is the skew that will happen, and it degrades to exactly the
old behaviour rather than to something worse.

**This is a behaviour change, not just a bugfix — anticipate it.** If you set
`COMMY_SUBSCRIBE` outside the plugin and have been silently overridden, your
value starts taking effect once both halves are in place. A seat you have been
running as effectively mentions-only will begin receiving the channel and topic
traffic you originally configured. That is the correct behaviour and the reason
for the fix, but it is a change in what lands in your seats' context, so check
that the value you set is still the value you want.

**Boot now says what it subscribed to.** When the boot-time subscribe set is
non-empty, the server logs the applied targets in the same token vocabulary you
wrote them in:

```
commy plugin: applied 2 boot-time subscribe target(s): myproject, general/standup
```

Compare that line against your config, character for character. No line means no
boot-time subscriptions were applied. It is deliberately silent on the empty
set rather than warning — a seat that wants no subscriptions and a seat whose
value was destroyed upstream are indistinguishable by then, so a warning would
fire on the majority of healthy interactive boots while telling the one broken
seat nothing actionable.

**Sticky thread subscriptions now survive a resume.** Posting or reacting into a
thread subscribes you to it. That intent was held in memory only, while an
intent acquired by *asking* (the explicit `subscribe` tool) was persisted — so a
resumed seat came back silently deaf to threads it believed it was still in. The
store was never wrong about what it held; it simply never held them. Sticky
engagement now persists on the same sinks as an explicit subscribe, and restore
honours it.

## Upgrading

**Publishing this release delivers half the fix. The other half needs the plugin
pin.** The two artefacts ship on different supply paths:

| half | artefact | ships via | moves when |
|---|---|---|---|
| server reads suffixed-first | `@codeforbreakfast/commy-mcp` | npm | next seat start (`npx` resolves `latest`, unpinned) |
| manifest writes the suffixed key | the plugin | marketplace ref | you bump the pin and run `/plugin` |

Take the server alone and nothing changes for you: the old manifest still writes
the bare name empty, so the new server's fallback never finds a suffixed value
to prefer and reads the same clobbered empty string as before. **The clobber
stops when both are loaded**, not when this release publishes. If you track the
marketplace ref, move it to `commy-v0.22.0` and confirm the boot line above
before you treat the fix as live.

**Update the server to ≥0.22.0 before or alongside the plugin, never after.**
Compatibility between the two halves runs one way only. The new manifest writes
*only* the suffixed key — the bare `COMMY_SUBSCRIBE` is no longer written at
all — so:

- **New server, old manifest** — safe. The suffixed key is absent, the server
  falls back to the bare name, and you get exactly today's behaviour. Neither a
  fix nor a regression; this is the state you sit in between this publish and
  your pin bump.
- **New manifest, old server** — **you lose working configuration.** The 0.21.0
  server reads a bare key nobody writes any more, finds nothing, and boots
  deaf. Any `COMMY_SUBSCRIBE` or `COMMY_CATCHUP_WINDOW_SECONDS` you supply
  through the plugin's config prompts silently stops taking effect.

Normally the server cannot lag, because `npx -y` resolves npm `latest` at every
seat start while the plugin moves only on an explicit pin bump. **Setting
`COMMY_NPM_MIN_RELEASE_AGE` breaks that assumption** — and it is the one knob
that does. The marketplace plugin ref is a github ref and is not gated by npm;
the server is. So an operator running a release-age soak takes the new manifest
immediately and stays on the old server for the whole soak window, with their
plugin-supplied subscriptions dropped for its duration. If you use a soak,
either clear it for this upgrade or delay the pin bump until your server has
actually moved past `0.22.0`, and use the boot line above to confirm which one
you are running.

**`npm_config_min_release_age` is unchanged and still written to the bare
name.** The `COMMY_NPM_MIN_RELEASE_AGE` user-config field feeds npm's own
variable, which is not part of commy's key space and has not been moved behind
the suffix. If you set `npm_config_min_release_age` globally and also enable the
plugin without supplying that field, the plugin still overrides it with empty.

**If you never set `COMMY_*` outside the plugin, there is nothing to do.**
Configure through the plugin prompts as before; the suffix is internal and you
will not see it.
