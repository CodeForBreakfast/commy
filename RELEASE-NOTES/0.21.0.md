commy plugin: release 0.21.0

Minor release of the commy plugin, **0.20.0 → 0.21.0**.

This release changes the `COMMY_SUBSCRIBE` token grammar and **rejects three
token forms that previous versions accepted**. If you set `COMMY_SUBSCRIBE`, read
**Upgrading** before you take this release — an un-migrated value fails the boot.

## Highlights

**Subscribe targets are bare paths.** A subscription names a place, and the
grammar now says that in one position:

```
myproject                 # the whole channel
myproject/general         # one topic in it
new-topics:myproject      # a rule over that channel
```

`new-topics:` keeps its prefix, and that is the point of the change rather than
an exception to it. It is not a different *place*, it is a different *rule* about
which messages in a place reach you — first-message-of-each-new-topic only. The
old grammar mixed the two axes: `channel:` and `thread:` were pure scope and
carried no information a bare path doesn't, while `new-topics:` was scope plus a
rule. Now a bare path is scope and a prefix is a modifier, and each position
means exactly one thing.

This applies everywhere the grammar is spoken: the `COMMY_SUBSCRIBE` env var, and
the `target` argument of the `subscribe` and `unsubscribe` MCP tools.

**Three retired forms are now errors rather than silence.** `channel:<name>`,
`thread:<channel>/<thread>`, and the `mentions` token are rejected with a message
naming what to write instead.

Rejecting them is the substance of the change, not tidiness. Channel names are
only validated for non-emptiness, so under the new grammar a surviving
`channel:myproject` would parse *successfully* — as a channel literally named
`channel:myproject`. The seat would boot cleanly, register a narrow against a
channel nobody posts to, and hear nothing, with no error anywhere for the
operator to find. A loud boot failure at the moment of misconfiguration is the
whole reason these are parse errors.

`mentions` was retired in 0.20.0 for a different reason — mentions of a bot
arrive unconditionally and never needed a token — and was accepted-and-ignored
there so that existing configs kept booting. That grace period ends here, and it
ends deliberately: all three dead forms go in one breaking version rather than
trickling out over several.

## Upgrading

**If you set `COMMY_SUBSCRIBE`, migrate it before upgrading.** Drop the `channel:`
and `thread:` prefixes, and delete any `mentions` token:

```diff
- COMMY_SUBSCRIBE=channel:myproject,thread:myproject/general,mentions
+ COMMY_SUBSCRIBE=myproject,myproject/general
```

`new-topics:myproject` is unchanged. An un-migrated value fails the boot with
`invalid subscribe token "…" — retired token form — write …`, which names the
replacement for the token it rejected.

**If you leave `COMMY_SUBSCRIBE` unset, there is nothing to do.** A fresh install
carries no default tokens, and mentions of your bot arrive without any
subscription.

**Agents that call `subscribe` / `unsubscribe` need no change.** The tool
descriptions carry the new grammar, so a connected agent reads the current forms
from the tool list.

Seats that persisted a `mentions` entry to their on-disk subscription store are
unaffected — that state is still read and discarded, so restoring a saved narrow
set does not fail on the dead entry.
