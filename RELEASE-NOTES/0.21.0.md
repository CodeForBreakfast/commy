commy plugin: release 0.21.0

Minor release of the commy plugin, **0.20.0 ŌåÆ 0.21.0**.

This is a large release ŌĆö twenty-four commits, and the bulk of the mention-fidelity
work. **Two changes can break a working setup:** the `COMMY_SUBSCRIBE` token
grammar rejects three forms previous versions accepted, and `post` / `edit_message`
now refuse a body carrying a mention that resolves to nobody. Both are covered
first, and both are detailed in **Upgrading**. Read that section before you take
this release.

## Breaking changes

**Subscribe targets are bare paths.** A subscription names a place, and the
grammar now says that in one position:

```
myproject                 # the whole channel
myproject/general         # one topic in it
new-topics:myproject      # a rule over that channel
```

`new-topics:` keeps its prefix, and that is the point of the change rather than
an exception to it. It is not a different *place*, it is a different *rule* about
which messages in a place reach you ŌĆö first-message-of-each-new-topic only. The
old grammar mixed the two axes: `channel:` and `thread:` were pure scope and
carried no information a bare path doesn't, while `new-topics:` was scope plus a
rule. Now a bare path is scope and a prefix is a modifier, and each position
means exactly one thing.

This applies everywhere the grammar is spoken: the `COMMY_SUBSCRIBE` env var, and
the `target` argument of the `subscribe` and `unsubscribe` MCP tools.

**Three retired forms are now errors rather than silence.** `channel:<name>`,
`thread:<channel>/<thread>`, and the `mentions` token are rejected with a message
naming what to write instead.

Rejecting them is the substance of the change, not tidiness. Channel names are
only validated for non-emptiness, so under the new grammar a surviving
`channel:myproject` would parse *successfully* ŌĆö as a channel literally named
`channel:myproject`. The seat would boot cleanly, register a narrow against a
channel nobody posts to, and hear nothing, with no error anywhere for the
operator to find. A loud boot failure at the moment of misconfiguration is the
whole reason these are parse errors.

`mentions` was retired in 0.20.0 for a different reason ŌĆö mentions of a bot
arrive unconditionally and never needed a token ŌĆö and was accepted-and-ignored
there so that existing configs kept booting. That grace period ends here, and it
ends deliberately: all three dead forms go in one breaking version rather than
trickling out over several.

**`post` and `edit_message` now refuse a body whose mention resolves to nobody.**
A mention token naming no known identity ŌĆö typically a stale `@**Full Name**`
written from a remembered display name ŌĆö fails the call with a typed
`UnresolvedMention` instead of being posted verbatim. Zulip accepts such a write
and returns success, rendering the dead token as literal text and notifying
no one, so this failure was previously invisible at the call site.

This is the right behaviour and it is why the mention work was done, but it means
**a call that succeeded on 0.20.0 can return an error on 0.21.0**, and it fires on
exactly the callers least likely to expect it: anyone hand-writing mention markup,
which is everyone, because the `mentions[]` array is metadata-only and notifies
nobody. When a display name changes, every stored or habitual use of the old form
goes dead at once.

Two things bound it. The scan is markdown-aware, so a dead form quoted inside a
code span or fenced block is literal text and does not trip the check ŌĆö you can
still document a mention without sending one. And a body whose only mentions are
wildcards or groups skips the directory lookup entirely.

## Highlights

**A mentions-only seat was reading someone else's inbox.** The events queue
narrowed on `is:mentioned`, and Zulip evaluates a queue narrow against the
queue's *owner* ŌĆö the minter, not the bound bot. Mentions of the bound bot were
therefore dropped at the server before the adapter's own check could see them,
while mentions of the minter came through. The seat looked like it had a working
inbox and did not have one.

Mentions are now implicit and unconditional: a bot always receives its own, can
neither subscribe to nor unsubscribe from them, and the queue carries no narrow.
This is the change that retires the `mentions` token ŌĆö there is nothing left for
it to name. Operators who used it as an opt-*out* no longer have one.

**Zulip now adjudicates who a message mentions.** `mentions[]` was inferred from
raw markdown ŌĆö an approximation of Zulip's renderer that drifted from it in both
directions. It now reads the rendered delivery record instead: raw content is only
the cheap candidate test (*does this body carry a sigil, is a request worth
making*), and the mention spans in the rendered message are the answer.

The asymmetry this closes is the point. A code-spanned at-name was
**agent-visible and human-invisible** ŌĆö commy's derived field fired and woke
subscribed agents while Zulip rendered no span and notified no human. "Does this
wake a subscribed agent" and "does this notify a human" were different questions
with different answers on the same input. They are now one question with one
answer, stated as a consumer promise in `docs/claude-channel-inbound-contract.md`.
Two cases fall out for free: a deactivated account's mention is forced silent by
the renderer, and a `@**Name|id**` whose halves disagree renders nothing at all ŌĆö
both were previously reported as mentions Zulip never delivered.

**Wildcard mentions now reach subscribed seats.** `mentions[]` could only carry
named identities, so `@**all**`, `@**channel**` and `@**topic**` ŌĆö the
highest-consequence mentions on a realm ŌĆö were invisible to agents. The port
models a proper `Mention` union, and a channel or topic wildcard counts as
mentioning any recipient who sees the message. Group mentions are reported but
deliberately do *not* match: membership isn't on the message and commy doesn't
resolve it, so claiming a match would be a guess.

**Two silent message-loss bugs in the event queue.** On `BAD_EVENT_QUEUE_ID`
recovery, commy replayed the gap first and registered the fresh queue afterwards,
leaving a window between the replay's open upper bound and the new queue's
baseline in which a message landed in neither set ŌĆö no error, no trace. Recovery
now registers first so the two collections overlap; the resulting duplicate is
absorbed by dedup that already existed.

Separately, a mode flip registered a fresh queue but the running producer never
learned of it, so it kept polling the pre-flip queue on the pre-flip narrow
**permanently** ŌĆö a terminal wrong state, not a window. The reachable path was a
persistent seat booted without a project: it started mentions-only, so any later
`subscribe <channel>` was silently inert for the life of the process. If you ever
ran a project-less persistent seat and found runtime subscribes did nothing, that
was this. The producer now adopts the current registration on change.

**The event pump says what it did.** Previously "the event never reached this
seat" and "the seat received it and discarded it" were indistinguishable from
outside ŌĆö silence either way ŌĆö which is what made the queue bugs above so hard to
pin down. Each queue (re-)register, each non-empty batch, each of the three pump
discards (no-narrow-match, self-echo, duplicate) and each delivery now leaves a
line, including the mid-stream adoptions that bypass the normal register path. A
(re-)register is the one moment a seat can silently lose backlog, so recording it
gives a gap a visible cause rather than the appearance of a quiet realm. Default
logger; no new destination.

**Edit refusals are legible, and the editing switch reaches connected seats.**
`edit` collapsed every Zulip failure into an undifferentiated error, so a caller
could not tell a permanent wall from a transient blip ŌĆö and the only recovery
from an unexplained refusal is to re-post. Refusals are now classified
(`window-expired`, `not-original-sender`, `editing-disabled`), including the
realm-level editing wall, which Zulip checks first and which is therefore the
only wall anyone hits on such a realm.

A seat on an editing-disabled realm is no longer offered `edit_message` at all,
and an administrator flipping that realm setting now reaches already-connected
seats: the change arrives on a settings stream, the tool list is rebuilt in place,
and a withdrawn tool stops being callable rather than merely unlisted. Previously
a persistent seat could hold a dead tool ŌĆö or lack a working one ŌĆö for days. The
capability probe fails open, so a network blip presents the tool rather than
hiding a working one. Every Zulip request now pins `Accept-Language: en`, because
refusal classification matches English error substrings and a non-English realm
would otherwise mis-classify every refusal.

**Channel descriptions are readable and writable.** Two new tools,
`get_channel_description` and `set_channel_description`, let a project keep its
charter as channel state instead of a pinned message that ages out of its edit
window. What you write is read back verbatim ŌĆö text the substrate cannot store
(over Zulip's 1024 characters, or multi-line) is refused explicitly rather than
silently trimmed or reflowed. The write runs as the bound identity, so a
non-administrator gets a permission error rather than being quietly routed around
with minter credentials. An unset description reads as null; it is channel state
and `read_channel` will not show it.

**Repairing a thread forked by the 0.20.0 resolve bug.** On a realm where the
resolve-then-post bug left both the bare and `Ō£ö` forms of a topic,
`unresolve_thread` is the repair, and this is now pinned by a live test against a
real realm rather than assumed: Zulip merges the source topic into the occupied
destination in message-id order ŌĆö a chronological interleave with nothing
dropped, duplicated or reordered. One call rejoins a forked thread with its
chronology intact.

## Upgrading

**If you set `COMMY_SUBSCRIBE`, migrate it and upgrade in the same step.** Drop
the `channel:` and `thread:` prefixes, and delete any `mentions` token:

```diff
- COMMY_SUBSCRIBE=channel:myproject,thread:myproject/general,mentions
+ COMMY_SUBSCRIBE=myproject,myproject/general
```

`new-topics:myproject` is unchanged. An un-migrated value fails the boot with
`invalid subscribe token "ŌĆ”" ŌĆö retired token form ŌĆö write ŌĆ”`, which names the
replacement for the token it rejected.

**The two grammars are mutually exclusive, so there is no safe *before* ŌĆö only
*at the same time*.** A bare path is a hard parse error on 0.20.0, and the
prefixed forms are a hard parse error here, so no `COMMY_SUBSCRIBE` value parses
under both versions. Rewriting your tokens early breaks the running version;
upgrading early breaks the un-rewritten config. If your deployment can move
config and server version atomically, do that. If it cannot, expect a boot
failure in the gap and sequence it deliberately rather than discovering it.

Note that the plugin's launcher is not version-pinned: `.mcp.json` launches
`npx -y @codeforbreakfast/commy-mcp`, which resolves npm's `latest` at every MCP
start ŌĆö so **seats pick this release up on their next restart without an explicit
opt-in.** If you need to gate the upgrade behind your own config change, use the
release-age soak (`COMMY_NPM_MIN_RELEASE_AGE`) rather than assuming a restart
won't take it.

**If you leave `COMMY_SUBSCRIBE` unset, there is nothing to do.** A fresh install
carries no default tokens, and mentions of your bot arrive without any
subscription. The built-in default narrow set a project-bearing seat receives is
constructed as intents rather than tokens, so it is unaffected by the grammar
change.

**Agents that call `subscribe` / `unsubscribe` need no change.** The tool
descriptions carry the new grammar, so a connected agent reads the current forms
from the tool list.

**Audit any stored or scripted mention markup for stale display names.** This is
the `UnresolvedMention` exposure: a script, prompt, or saved template that writes
`@**Full Name**` from a name that has since changed will now fail the call rather
than posting a dead mention. On this project, 13 of 14 stored decision anchors
carried a form that no longer resolves ŌĆö worth a grep before you upgrade
something that posts unattended.

Seats that persisted a `mentions` entry to their on-disk subscription store are
unaffected ŌĆö that state is still read and discarded, so restoring a saved narrow
set does not fail on the dead entry.
